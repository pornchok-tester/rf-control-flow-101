🔄 ทบทวนบทที่แล้ว

List ที่ไม่มี item เลย — ถ้าเอาไปใช้กับ FOR จะเกิดอะไรขึ้น? แล้วถ้าต้องการให้ fail ต้องทำยังไง?

ดูเฉลย
FOR จะไม่ทำงานเลย — เงียบๆ และผ่านไป ไม่มี error | ถ้าต้องการ fail ต้องใช้ Should Not Be Empty ก่อน loop
บทที่ 5

Loop + Selenium หน้างานจริง

✅ SeleniumLibrary 6.x

🎯 บทนี้จะได้อะไร

  • รอ loading / spinner หายด้วย WHILE แทน Sleep แบบ hardcode
  • วน element บนหน้าเว็บที่จำนวนไม่แน่นอนด้วย Get WebElements
  • retry action ที่บางครั้ง click ไม่ติดครั้งแรก
  • วน pagination จนหมดทุกหน้า
keyword ใหม่ในบทนี้
Get WebElements locator — คืน list ของ element ทุกตัวที่ match locator ถ้าไม่เจอเลยคืน list ว่าง (ไม่ fail)
Get Text locator_or_element — คืน text ที่อยู่ใน element นั้น
Run Keyword And Return Status keyword args — รัน keyword แล้วคืน True ถ้าผ่าน, False ถ้า fail — ไม่หยุดการทดสอบ
Pattern 1

รอ Loading / Spinner ด้วย WHILE

Sleep 5s แบบ hardcode คือปัญหา — ถ้าโหลดเร็วก็เสียเวลาเปล่า ถ้าโหลดช้ากว่านั้นก็ยัง fail อยู่ดี WHILE แก้ได้ตรงๆ โดยเช็คทุก 0.5 วินาทีจนกว่า spinner จะหาย

WHILE รอ spinner หายก่อนทำต่อ

keyword นี้ใช้ซ้ำได้ทั่วทั้ง suite — เรียกทุกครั้งที่ต้องรอก่อนทำต่อ:

*** Keywords ***
Wait For Spinner To Disappear
    [Arguments]    ${timeout}=30 seconds
    WHILE    True    limit=${timeout}
        ${loading}=    Run Keyword And Return Status
        ...    Element Should Be Visible    css=.loading-spinner
        IF    not ${loading}    BREAK
        Sleep    0.5s
    END

*** Test Cases ***
Submit Order
    Fill Order Form
    Click Element    css=button#submit
    Wait For Spinner To Disappear    # รอ spinner หายก่อน
    Element Should Be Visible    css=.order-success
❌ วิธีที่พบบ่อย
Click Element    css=button#submit
Sleep    5s    # hardcode — ไม่รู้ว่าพอหรือเปล่า
Element Should Be Visible    css=.order-success
✅ วิธีที่ถูก
Click Element    css=button#submit
Wait For Spinner To Disappear
Element Should Be Visible    css=.order-success
Pattern 2

วน element ที่ไม่รู้ว่ามีกี่ตัว

เมื่อหน้าเว็บ render รายการที่จำนวนขึ้นอยู่กับ data เช่น notification, ผลการค้นหา, รายการสินค้า — Get WebElements ดึง element ทั้งหมดมาเป็น list แล้วใช้ FOR วนผ่านได้เลย

Get WebElements + FOR — ตรวจทุก row ใน table
*** Keywords ***
Verify All Products Have Name
    @{cards}=    Get WebElements    css=.product-card .name
    Should Not Be Empty    ${cards}    ไม่เจอ product card เลย

    FOR    ${card}    IN    @{cards}
        ${name}=    Get Text    ${card}
        Should Not Be Empty    ${name}    เจอ card ที่ไม่มีชื่อ
    END
จำไว้: Get WebElements ไม่ fail ถ้าไม่เจอ element — คืน list ว่างเงียบๆ ถ้าไม่ใส่ Should Not Be Empty ก่อน loop test จะผ่านทั้งที่ไม่ได้ตรวจอะไรเลย
Pattern 3

Retry Action ที่บางครั้ง Click ไม่ติด

บาง element อยู่บนหน้าเว็บแล้ว แต่ยังคลิกไม่ได้ เช่น button ที่กำลัง animate เข้ามา หรือ dropdown ที่ต้อง hover ก่อน แทนที่จะ fail ทันที ให้ลองซ้ำก่อนสัก 3 ครั้ง

FOR + BREAK — retry 3 ครั้งก่อน fail
*** Keywords ***
Click With Retry
    [Arguments]    ${locator}    ${max_retry}=3
    FOR    ${attempt}    IN RANGE    ${max_retry}
        ${ok}=    Run Keyword And Return Status
        ...    Click Element    ${locator}
        IF    ${ok}    BREAK
        Sleep    1s
    END
    Should Be True    ${ok}
    ...    Click ไม่สำเร็จหลังลอง ${max_retry} ครั้ง: ${locator}

*** Test Cases ***
Submit Order
    Open Order Page
    Fill Order Form
    Click With Retry    css=button#submit-order
    Wait Until Element Is Visible    css=.order-success
Pattern 4

วน Pagination จนหมดทุกหน้า

ใช้ตอนที่ต้องตรวจข้อมูลทุกหน้า เช่น verify ว่า item ที่สร้างไปปรากฏใน list — ไม่รู้ว่าอยู่หน้าไหน ต้องวนจนเจอหรือจนหมด

WHILE วน Pagination — หยุดเมื่อไม่มีหน้าถัดไป
*** Keywords ***
Find Order On Any Page
    [Arguments]    ${order_id}
    WHILE    True    limit=20
        Wait Until Element Is Visible    css=.order-list

        # ตรวจหน้าปัจจุบัน
        ${found}=    Run Keyword And Return Status
        ...    Element Should Contain    css=.order-list    ${order_id}
        IF    ${found}    BREAK

        # เช็คว่ามีหน้าถัดไปไหม
        ${has_next}=    Run Keyword And Return Status
        ...    Element Should Be Visible    css=button.next-page:not([disabled])
        IF    not ${has_next}    BREAK

        Click Element    css=button.next-page
    END

    Should Be True    ${found}    ไม่พบ order ${order_id} ในทุกหน้า
ตั้ง limit ไว้เสมอ — ถ้า pagination มี bug แล้วปุ่ม Next ไม่ disabled ทั้งที่หมดแล้ว WHILE จะวนไม่หยุด limit=20 ทำให้ test fail แทนที่จะค้างตลอด

🧠 เช็คความเข้าใจ

ข้อ 1. Run Keyword And Return Status ต่างจาก keyword ปกติยังไง? ทำไมถึงต้องใช้ใน pattern เหล่านี้?

ดูเฉลย
keyword ปกติเมื่อ fail จะหยุดการทดสอบทันที | Run Keyword And Return Status จะ "จับ" ผลนั้นไว้แล้วคืน True/False แทน ทำให้ใช้ใน IF ได้โดยไม่หยุดทดสอบ — เหมาะกับสถานการณ์ที่ fail เป็นเรื่องปกติ เช่น spinner อาจยังไม่หาย หรือ click อาจยังไม่ติด

ข้อ 2. ถ้า Get WebElements ไม่เจอ element เลย แล้วไม่ได้ใส่ Should Not Be Empty ไว้ก่อน FOR — test จะเป็นยังไง?

ดูเฉลย
test ผ่าน — FOR ไม่ทำงานเลย ไม่มี error แต่นั่นหมายความว่าไม่ได้ตรวจอะไรเลย ผลคือ test ไม่มีความหมาย ดูเหมือนผ่านแต่จริงๆ แล้ว miss ปัญหาไปหมด

ข้อ 3. Pattern Retry กับ Pattern WHILE Spinner ต่างกันยังไง? แต่ละ pattern แก้ปัญหาอะไร?

ดูเฉลย
WHILE Spinner — รอให้ระบบพร้อมก่อนทำต่อ ไม่รู้ว่าจะรอนานแค่ไหน | Retry — ลอง action ซ้ำในจำนวนครั้งที่จำกัด เพราะ action นั้นบางครั้งทำไม่ได้ครั้งแรก ต่างกันตรงที่ Spinner รอ passive / Retry ลอง active
